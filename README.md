# Tideway marketplace

Independent pilot website for matching landlords and businesses with cleaning professionals. It does not use Polsia or Polsia credits.

## Run locally

Use Node.js 20 or newer:

```text
npm start
```

Then open `http://127.0.0.1:4173`.

The local NDJSON pilot remains dependency-free at runtime while `MARKETPLACE_ENABLED=false`. A production/staging marketplace deployment must use the pinned pnpm toolchain and reviewed lockfile:

```text
node tools/check-dependency-lock.mjs
pnpm install --frozen-lockfile --ignore-scripts
```

The manifest pins `pg` 8.22.0 and the reviewed Stripe Node SDK 22.1.1 exactly; the gate locks the entire dependency graph by SHA-256 plus registry integrity values. Do not use an unlocked install or commit `node_modules`.

Before any production process starts, configure the reviewed HTTPS proxy, off-source private data directory and protected Administrator boundary, then run `pnpm run preflight:production`. The server repeats this gate before listening. A safe public-site deployment can keep both marketplace and payments false without inventing database credentials; authenticated marketplace promotion remains a separate managed-staging step. See [Production deployment gate](docs/PRODUCTION_DEPLOYMENT.md).

For a phone on the same trusted Wi-Fi, run `npm run start:phone`, find the computer's private IPv4 address, and open `http://<computer-ip>:4174/` on the phone. The main control desk remains on `http://127.0.0.1:4173/admin`; both its HTML shell and data APIs require an admin key through the Wi-Fi address. This is a local-network preview only—do not expose port 4174 through a router or public tunnel.

## Marketplace design preview

The homepage now links to a truthful, interactive marketplace preview. Open `/marketplace-preview?screen=profile` to inspect the Landlord-facing Cleaner profile or `/marketplace-preview?screen=tracking` to try the Landlord/Cleaner journey, arrival, cleaning-progress and completion states. These screens use labelled sample UI only: they do not create a booking, read browser location or imply that a Cleaner, rating, price, verification or active job exists.

To exercise the real journey and cleaning handoff locally, open `http://127.0.0.1:4173/tracking-test` directly. Create a private 30-minute session, open its separate Landlord viewer in another tab, consent in the Cleaner controller and press **Start real journey**. After the first location point, **I have arrived** deletes that point and unlocks a real server-side sample cleaning checklist. The Cleaner can start cleaning, complete or reset tasks, report and resolve issues, and finish only when all four tasks are complete; every update streams to the read-only Landlord view. This lab is not a booking. It writes no route or business record, uses no external map provider and clears the session on Delete, expiry or server shutdown. The Cleaner controller is loopback-only; the HTTP phone preview may be a viewer but cannot request mobile geolocation without trusted HTTPS. The homepage and public preview do not link to this lab, and production returns 404 for its page, script and APIs.

Confirmed marketplace bookings now have a real authenticated [active-job interface](docs/ACTIVE_JOB_UI.md) at `/bookings/:bookingId`, with `/tracking` and `/cleaning-progress` aliases. The Cleaner gets explicit-consent foreground journey sharing, arrival, start/pause/resume, role-bound task/issue updates, Landlord approval for unexpected work and a resolved-checklist finish gate. The Landlord receives participant-only durable live snapshots, ETA/current-point status, room progress, notes and protected visit details, then confirms finished work before submitting the booking's one verified review. An approved review can receive one final Cleaner response; pending moderation stays private and only approved rows affect public rating. No sample booking is inserted and no live point is sent to an external map provider. The route remains honestly unavailable until the detached PostgreSQL account runtime passes staging; the final HTTPS two-phone test remains open.

Phase 1 includes transaction-local RLS identity, opaque sessions/CSRF, scrypt passwords, role onboarding, single-use email verification/reset, persistent credential lockout, session revocation and reusable exact-origin/CSRF/role middleware. `/login` and `/signup` contain capability-gated Google and Facebook controls. Google requires provider-verified email; Facebook validates the App-bound identity but then requires Tideway's own single-use mailbox verification before first account creation. In `/settings`, password accounts use current-password step-up; social-only accounts reauthenticate through an exact connected provider subject before a short-lived ten-minute approval can connect another method. The browser approval is cleared when a connection starts. Removal requires proof through a method that remains, refuses the final method and revokes every active session. Provider flags remain hidden until the complete runtime, managed PostgreSQL staging, SMTP, final HTTPS domain and verified provider credentials are ready. The read-only domain verifier can then require an exact `google`, `facebook` or combined enabled state and prove each advertised start route without following its external redirect or disclosing flow material.

Public **Book a clean** actions now use the fixed `/signup?intent=book` account-first journey. A configured Google or Facebook provider retains that action in signed server state, creates or safely reuses the verified Tideway account, and continues a new user to preselected Landlord onboarding before opening the private Landlord workspace. A first-time Landlord immediately receives the property form; a returning Landlord with saved properties goes directly to the room-scan request form. Saving the request automatically opens and focuses its required private room scan; successful submission then shows a dedicated authenticated thank-you state with the request reference, scope counts, matching boundary and tracking/new-request actions. Email/password remains the fallback; its private verification and resend links carry the same fixed action so a new tab or device returns to booking sign-in instead of losing the journey. The browser stores only this allowlisted action for 30 minutes; arbitrary destinations are rejected, and an existing Cleaner-only account is never silently granted the Landlord role.

Phase 2 has started with the [cleaner directory privacy boundary](docs/CLEANER_DIRECTORY.md) and [real Cleaner profile interfaces](docs/CLEANER_PROFILE_UI.md): cleaner-owned validated profile reads/updates, deterministic completion/publish gating and a restricted search function for location, availability, rating, price, service, distance and verified status. `/cleaners` now renders only real API results and `/cleaner/profile` provides an authenticated, CSRF-protected mobile editor. Direct public service-area coordinates are closed and public projections discard contact/address/internal fields. Genuine profiles remain unavailable until PostgreSQL integration, account activation and real Cleaner onboarding pass.

The [landlord property privacy boundary](docs/PROPERTY_PRIVACY.md) now adds owner-bound profile/property services, validated saved checklists, AES-256-GCM entry-instruction encryption and booking-scoped exact-address/access projections. `/landlord/dashboard` is a real-account, fail-closed [Landlord workspace](docs/LANDLORD_DASHBOARD_UI.md): an authenticated Landlord can save private properties, turn speech into a concise room checklist and save an exact future request draft. It always sends `submit: false`, so it cannot match, invite, book or take payment. The internal [private S3 media boundary](docs/PRIVATE_JOB_MEDIA.md) now adds signed quarantine uploads, checksum/encryption headers, bounded decode and metadata-stripping JPEG output. Account photo/video capture remains closed until PostgreSQL, authenticated storage/RLS staging and genuine accounts are ready; the working `/request` pilot remains the real scan route meanwhile.

The [account marketplace HTTP runtime](docs/MARKETPLACE_HTTP_RUNTIME.md) now composes these modules behind an isolated native-Node controller with session, exact-origin, CSRF and server-side role checks, bounded JSON and safe errors. Its routes are tested but deliberately not attached to the live pilot. Exact `pg` 8.22.0, Nodemailer 9.0.3, matched AWS S3 SDK 3.1084.0 and Sharp 0.35.3 packages are integrity locked. Real PostgreSQL, SMTP, private-bucket/CORS/threat-model evidence, monitoring and genuine account-backed data are still required.

The authentication runtime also includes trusted session issuance, exact logout, logout-all and same-account rotation. Raw session and CSRF tokens never cross the repository boundary, optional client metadata is stored only as keyed hashes, and a failed privilege-changing rotation leaves the old session revoked. Expired rows are removed only through a bounded `SKIP LOCKED` maintenance function: the web role cannot delete sessions and the separately credentialed worker has no direct table access. The [default-off worker process](docs/WORKER_OPERATIONS.md) now schedules this and four other maintenance jobs without overlap, with private failure monitoring and privacy-safe health; it passed locally through the real restricted PostgreSQL 16.14 role and still requires managed staging before activation.

Email authentication handlers are now prepared behind the same detached runtime: generic signup, verification resend/confirmation, login, password-reset request/confirmation, logout and role onboarding. Replacement verification invalidates older unused links, email tokens travel only in URL fragments, and the internal [SMTP delivery](docs/SMTP_EMAIL_DELIVERY.md) enforces exact-origin text-only mail, strict TLS and bounded transport behavior. No account control appears on the live site before PostgreSQL, SMTP and HTTPS pass staging.

Google sign-in is also prepared behind that detached runtime. It uses authorization code flow with a signed ten-minute HTTP-only state cookie, nonce and PKCE, performs the code exchange only on the server, verifies Google's RS256 signature/JWKS, issuer, audience, expiry, nonce and `email_verified`, and then issues only Tideway's opaque session. It requests identity scopes only and does not retain Google access or refresh tokens. The button remains hidden until Google credentials and the complete runtime are attached.

## What works

- Functional journey-and-cleaning lab at `/tracking-test`: a loopback-only Cleaner controller creates a 30-minute in-memory session and a separate read-only Landlord viewer. Explicit consent starts real foreground browser `watchPosition()` updates; an authenticated stream delivers only the latest two-minute point, then arrival deletes it and unlocks four server-owned sample tasks. Only the Cleaner can start cleaning, complete/reset tasks, report an issue or finish, and the server rejects completion until every task is resolved. Delete/expiry/shutdown clears the session; no customer, Cleaner, booking or route-history file is written. The same-Wi-Fi phone page can be a viewer, while phone location control remains correctly blocked pending trusted HTTPS
- The marketplace tracking preview now begins journey/arrival at 0% cleaning, derives progress from four actual sample room tasks, keeps Landlord controls read-only, lets the Cleaner mark tasks or report a sample issue, blocks the primary Finish action until every task is resolved and labels before/after images as private placeholders rather than evidence that exists

- Truthful pre-launch geography: public copy says the limited local pilot is still being prepared and that coverage is verified before any quote or booking; it does not name London, claim applications are open or imply a service area while the founder-approved pilot postcode list is empty
- Uber-simple customer intake at `/request` that removes the marketing page and cleaner application, presents the complete server-validated request as three short stages—job, timing/access and contact—requires the first date the customer could accept so every new lead can enter exact-availability matching, keeps entries when moving back, keeps each new step title visible below the sticky phone header, gives immediate step-level UK-postcode, phone and access-secret feedback from the same rules used by the server, and only unlocks the required room scan after the request is created. The early request collects only a general access approach; door, gate, alarm and key-safe codes, passwords and exact hidden-key locations are rejected from the access, site-size and job-detail fields until the protected accepted-booking stage
- Customer request recovery keeps only allowlisted property, timing, safe access and contact entries in the current tab for 30 minutes and restores the exact form stage after a reload. Access secrets, privacy consent and the anti-spam field are never stored or restored; offline submission is stopped, success/expiry/discard clears the draft, and an interrupted submission reuses its exact random retry key so a reload cannot silently duplicate the lead
- Server-validated cleaning frequency carried from request through tracker, frozen customer quote, cleaner opportunity, booking audit, protected packs and recipient handoffs; weekly, fortnightly and turnover preferences remain planning context while every accepted quote confirms one dated visit only, never an automatic recurring schedule
- Focused cleaner intake at `/join` that removes the customer form and separates basic details, work/travel preferences and availability/confirmations into three short stages while preserving mandatory service selection, matchable postcode coverage, right-to-work confirmation and consent. The details step catches invalid UK postcodes and phone numbers immediately, while the work step explains the required coverage format and uses the same parser as the server to stop vague, unmatchable travel coverage before final submission. A professional introduction, languages and equipment plan may be completed together now or through the private tracker after the core application is saved. The application captures one validated exact first-available window and a successful submission continues automatically into its private tracker with a visible recovery button. An active applicant can keep adding non-overlapping future windows during review so supply does not become stale, but every update remains pending and unusable for matching until screening, current approval and a separate Tideway confirmation are complete; paused and closed applications cannot update availability
- Two-stage private Cleaner profile starter: an applicant may save the core application first, then add or update a 40–600 character professional introduction, one-to-ten deduplicated languages and an allowlisted equipment/products plan through the token-authorised tracker. Updates are retry-safe, append-only and merged into the founder view without exposing retry metadata. The tracker returns these fields only to that private applicant link. The server refuses to mark all screening checks complete—and refuses approval—until the full starter exists. Nothing is published, verified, priced or made matchable automatically
- Private live Cleaner application preview on the final onboarding step projects only the applicant's name, selected services, professional introduction, experience, deduplicated languages, equipment plan, declared work areas and first supplied window. An eight-item preview guide names incomplete presentation details as the applicant types; it is guidance, not the core-application submission gate. The preview deliberately excludes email, phone and home postcode, uses text-only rendering, treats vague postcode coverage as incomplete and remains labelled **Unverified · not published**; it never creates a public profile, rating, price, badge, approval, availability confirmation or job promise
- Cleaner application recovery keeps only allowlisted entries and service choices in the current tab for 30 minutes and restores the exact form stage after a reload. Right-to-work, privacy consent and the honeypot are never stored or restored. Offline submission is stopped clearly, each form request times out after 30 seconds without erasing entries, an interrupted exact submission reuses its random retry key after reload, and success, expiry or explicit discard clears the draft
- High-entropy private cleaner tracker created with each new application; the application token is carried in the browser fragment and removed before loading. It reports only evidence-backed receipt, screening, approval, exact-availability and opportunity-readiness stages, lets a screened approved cleaner submit retry-safe exact future windows for confirmation, never promises work and exposes no identity, contact, travel or screening-note data
- Retry-safe intake: customer requests, cleaner applications and room scans send random idempotency keys; concurrent or network-retried copies return the original reference instead of creating duplicate leads or scan versions, while changed details require a new key
- High-entropy private customer tracker created with each request; it follows the real journey from required room scan through quote, cleaner confirmation, protected booking, job progress and completion. It keeps the requested date and arrival window visible in UK-local terms while making clear that these remain preferences until an exact private quote is accepted. - Customer-owned pre-booking timing recovery lets the private tracker replace the requested date and arrival window with a validated future choice and a bounded reason. The original and every replacement remain append-only, exact retries do not duplicate history, matching/admin use only the latest customer decision, and a passed date produces an explicit re-date gate instead of a vague empty match. Reloading restores that text but deliberately requires every photo/video to be selected again and the checklist to be re-summarised; no media, email or private token enters the recovery draft. Offline status blocks submission without losing text, a selected-visual leave warning prevents accidental reload loss, and success, expiry or explicit discard clears the draft

- Photo/video-and-voice room scan with separate mobile controls for an immediate single rear-camera photo and existing-device photo/video selection. The landlord chooses one exact current room to unlock capture, every newly added visual receives that validated room automatically, repeated captures keep it, and every card remains editable for multi-room batches. Starting voice capture inserts a visible validated “In the [room].” marker into the saved transcript; changing the current room while listening inserts the next marker, so ordinary speech stays grouped per room without a hidden transient context and reload/freshness protection sees the exact source. It accepts up to ten room visuals, including at most two 30-second walkthrough videos, still requires a specific note for every visual, and automatically summarises speech and media notes into concise room-labelled checklist bullets. Keeping the current list leaves checklist freshness incomplete, an empty generated result never erases existing work, and source-fingerprint validation rejects a comparison if the speech or room notes changed again before application. Voice capture uses the same review path instead of silently replacing manual edits

- Live scan-readiness guidance tracks eight required items as the customer works: connected request details, one-to-ten room visuals, room labels and notes, instructions, concise tasks, room coverage, final-scope confirmation and media-sharing permission; it names each shown room still missing a cleaner task, the action explains every outstanding check when clicked, and the server independently repeats every security-critical check

## Before public launch

1. Add the legal operator name, trading address and verified contact details.
2. Decide the cleaner engagement model with UK employment/tax advice.
3. Confirm public liability and any other required insurance.
4. Approve pricing, cleaner pay, cancellation, complaint and re-clean rules.
5. Replace local file storage and single-process rate limiting with an encrypted production database, shared abuse controls, access monitoring and unique database constraints for submission retry keys.
6. Approve the inactive-enquiry and completed-booking media periods already enforced by the local retention desk, then document the production speech-recognition provider, encrypted photo storage and the complete record-retention schedule.
7. Configure the final public HTTPS website origin, verify domain ownership and the deployed pages, then record a short evidence summary naming the exact hostname and the verification date; local, IP-address, path-bearing, insecure or unsupported origins cannot generate dispatch handoffs.
8. Add transactional email/SMS only after the sending account is approved.
9. Complete a real pilot in one small service area before making broader coverage claims.

If the server is ever bound to a public interface, set a strong `ADMIN_KEY`. Local control-desk access is automatic only when the server, request hostname and network connection are all verified as loopback and no proxy headers are present. Set `ADMIN_REQUIRE_KEY=true` to require that key on localhost too; this is recommended on shared computers and required for any non-dedicated operator host.

All browser mutations require a same-origin `Origin` header. Admin API reads also require same-origin browser metadata, in addition to admin authorisation. A room scan can only be submitted from its private request-tracker link; knowing a request reference and email address is not sufficient and the scan response never returns the tracker token.

Private pilot records must not remain in a OneDrive, Dropbox, Google Drive or iCloud-synchronised project folder. Set `DATA_DIR` to an absolute local folder outside those services before real intake. On Windows, a suitable per-user starting point is a new restricted folder under `$env:LOCALAPPDATA\Tideway\data`; verify its access permissions and backup destination before use. Tideway prints a prominent privacy warning whenever it detects a cloud-synchronised private-data path. The control desk reports only the safe/unsafe state and provider—not the local path—and unsafe storage now fails the operating-rules launch gate. It never moves data automatically. The inactive-enquiry and completed-booking retention periods remain mandatory launch-readiness decisions.

Use `tools/relocate-data.ps1` to rehearse the off-sync data-folder change without writing anything. Its executed mode requires an explicit destination, a stopped-server confirmation and an exact typed phrase; it refuses project-internal, cloud-synchronised or non-empty destinations, verifies every copied file by SHA-256, rechecks that the source did not change during copying and never deletes the source. Follow the complete founder-controlled procedure in [RECOVERY.md](RECOVERY.md#private-data-relocation).

Set `TRUST_PROXY=true` only when Tideway is directly behind a trusted reverse proxy that removes client-supplied forwarding headers and writes exactly one client IP, and configure its immediate network in `TRUSTED_PROXY_CIDRS`. The local pilot ignores forwarded addresses by default. The detached marketplace now verifies the immediate peer before using a forwarding value and has an atomic PostgreSQL limiter with scope-separated private client hashes, but it still requires PostgreSQL 16 execution and a real two-instance proxy test before production. See [trusted client identity](docs/TRUSTED_CLIENT_IDENTITY.md).
