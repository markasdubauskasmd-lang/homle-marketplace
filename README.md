# Tideway marketplace

Independent pilot website for matching landlords and businesses with cleaning professionals. It does not use Polsia or Polsia credits.

## Run locally

Use Node.js 20 or newer:

```text
npm start
```

Then open `http://127.0.0.1:4173`.

For a phone on the same trusted Wi-Fi, run `npm run start:phone`, find the computer's private IPv4 address, and open `http://<computer-ip>:4174/` on the phone. The main control desk remains on `http://127.0.0.1:4173/admin`; requests to its data APIs through the Wi-Fi address still require an admin key. This is a local-network preview only—do not expose port 4174 through a router or public tunnel.

## Authenticated marketplace expansion

The current pilot remains working while Tideway is extended into a three-role account marketplace. Read [the architecture audit](docs/ARCHITECTURE_AUDIT.md), [phased implementation plan](docs/MARKETPLACE_IMPLEMENTATION_PLAN.md), [database setup boundary](docs/DATABASE_SETUP.md) and [authentication security boundary](docs/AUTHENTICATION_SECURITY.md). Phase 1 includes transaction-local RLS identity, opaque sessions/CSRF, scrypt passwords, verified-email social deduplication, role onboarding, single-use email verification/reset, persistent credential lockout, session revocation and reusable exact-origin/CSRF/role middleware. `/login` and `/signup` now provide honest branded entry states with no non-working controls; provider flags cannot become public until the HTTP runtime is actually composed. User accounts remain disabled pending real PostgreSQL, migration tests, SMTP, cryptographic provider adapters, verified credentials and HTTPS deployment.

Phase 2 has started with the [cleaner directory privacy boundary](docs/CLEANER_DIRECTORY.md): cleaner-owned validated profile updates, deterministic completion/publish gating and a restricted search function for location, availability, rating, price, service, distance and verified status. Direct public service-area coordinates are closed and public projections discard contact/address/internal fields. No cleaner directory page is exposed because the local pilot currently has no genuine account-backed cleaner supply and PostgreSQL integration tests are still outstanding.

## What works

- Truthful pre-launch geography: public copy says the limited local pilot is still being prepared and that coverage is verified before any quote or booking; it does not name London, claim applications are open or imply a service area while the founder-approved pilot postcode list is empty
- Uber-simple customer intake at `/request` that removes the marketing page and cleaner application, presents the complete server-validated request as three short stages—job, timing/access and contact—keeps entries when moving back, keeps each new step title visible below the sticky phone header, gives immediate step-level UK-postcode and phone feedback from the same rules used by the server, and only unlocks the required room scan after the request is created
- Server-validated cleaning frequency carried from request through tracker, frozen customer quote, cleaner opportunity, booking audit, protected packs and recipient handoffs; weekly, fortnightly and turnover preferences remain planning context while every accepted quote confirms one dated visit only, never an automatic recurring schedule
- Focused cleaner intake at `/join` that removes the customer form and separates basic details, work/travel preferences and availability/confirmations into three short stages while preserving mandatory service selection, matchable postcode coverage, right-to-work confirmation and consent. The details step catches invalid UK postcodes and phone numbers immediately, while the work step explains the required coverage format and uses the same parser as the server to stop vague, unmatchable travel coverage before the applicant reaches final submission. The application captures one validated exact first-available window, carries it into the private tracker and prefills the founder workflow, but keeps it unavailable to matching until screening, approval and separate verification. A successful application continues automatically into its private tracker with a visible recovery button
- High-entropy private cleaner tracker created with each new application; the application token is carried in the browser fragment and removed before loading. It reports only evidence-backed receipt, screening, approval, exact-availability and opportunity-readiness stages, lets a screened approved cleaner submit retry-safe exact future windows for confirmation, never promises work and exposes no identity, contact, travel or screening-note data
- Retry-safe intake: customer requests, cleaner applications and room scans send random idempotency keys; concurrent or network-retried copies return the original reference instead of creating duplicate leads or scan versions, while changed details require a new key
- High-entropy private customer tracker created with each request; it follows the real journey from required room scan through quote, cleaner confirmation, protected booking, job progress and completion. It keeps the requested date and arrival window visible in UK-local terms while making clear that these remain preferences until an exact private quote is accepted. Before any booking exists, the customer can explicitly close the enquiry with a reason; the append-only decision is retry-safe and immediately closes scan, matching, quote, cleaner-opportunity, media and booking paths without deleting evidence
- Continuous private handoff from a completed request into the required room scan: the scan opens automatically with the reference and fragment token, removes that token before its first request, hides the repeated email field and retains a visible recovery button if navigation is interrupted. The private customer tracker can resume the same scan on another device through its fragment token while the server rechecks that token against the exact request
- Customer and cleaner tracker links keep their token in the browser fragment, remove it before the first server request and return no customer contact/access data, cleaner identity/pay/travel data, screening notes or tracker authorisation token
- Photo/video-and-voice room scan with separate mobile controls for an immediate single rear-camera photo and existing-device photo/video selection; it accepts up to ten room visuals, including at most two 30-second walkthrough videos, requires a deliberate room label and specific note for every visual, and automatically summarises speech and media notes into concise room-labelled checklist bullets. Natural follow-on sentences retain their current room, passive phrases such as “the floor needs mopping” become direct cleaner actions, and equivalent repeated instructions collapse to one bullet; saving remains blocked until every shown room has a task
- Canonical multi-room scope labels keep numbered bedrooms and bathrooms aligned between photos and speech, while common spoken names such as “lounge” and “WC” map to the same cleaner-facing room labels
- Required customer scope-completeness confirmation: the scan cannot be stored until the customer confirms the final concise checklist includes every task they want quoted; the server records the confirmation time and review, matching, private decisions and booking audit all retain the gate
- The customer’s scope confirmation is tied to the exact current speech, task list, room labels and media notes; changing any of them clears the checkbox and requires a fresh review before submission
- Concise cleaner tasks are tied separately to the speech and room-media notes that generated them; later source changes mark the bullets out of date and block submission until the customer summarises again, while manual review edits to the current bullets remain allowed
- Live scan-readiness guidance tracks eight required items as the customer works: connected request details, one-to-ten room visuals, room labels and notes, instructions, concise tasks, room coverage, final-scope confirmation and media-sharing permission; it names each shown room still missing a cleaner task, the action explains every outstanding check when clicked, and the server independently repeats every security-critical check
- Genuine room-scan submission shows per-visual and saving progress, enforces two-video/30-second/15 MB video limits plus the existing photo limits, has bounded media-preparation and request timeouts, and restores an actionable button with the customer's entries preserved if an upload fails
- A successful scan now leaves the long form for a dedicated thank-you page showing the saved scan reference, three clear next steps and the protected customer tracker; the tracker token remains in a browser fragment and is removed immediately
- Audited human review decisions for room scans; approval requires the founder to load every private visual, confirm each visual and its room note individually, and separately confirm the concise checklist was reconciled against all spoken notes, room notes and shown rooms. The server requires the exact submitted visual-ID set, stores a versioned evidence trail and the integrity audit detects incomplete or altered evidence; missing evidence fails closed across matching, quoting and booking
- Structured scan review requires a 0.5–24 hour human scope estimate, medium/high confidence and an evidence note; proposal hours default to the higher of this estimate or the founder minimum
- Price-sensitive scan detection flags oven/fridge interiors, inside storage, windows, linen/laundry, carpet or upholstery work, waste removal, outdoor areas and wall/ceiling washing; the customer sees the warning while reviewing the scan and the reviewer must explicitly confirm every detected item inside the cleaning-time estimate before matching or quoting can proceed
- Confirmed price-sensitive items are frozen into the customer quote and cleaner opportunity, shown in both message drafts and the customer tracker, and retained in both protected booking packs so scope cannot silently disappear between scan and job day
- Cleaner application form with server-side validation
- Audited seven-check cleaner screening record; approval, matching, proposals and bookings remain blocked until every check is confirmed
- Local private submission files in `data/`
- Private local control desk at `/admin` with lead filters, status tracking, internal notes and next-action dates
- Non-destructive private-data integrity desk checks all 18 append-only record files plus launch configuration at startup, on demand and before every write; it reports safe file/line/reference diagnostics, detects malformed records, duplicate primary references, disconnected request-to-booking links and invalid or inconsistent configuration/proposal/booking/outcome/adjustment economics, and fails closed without changing the damaged data
- Prioritised founder-action dispatch queue derived from recorded scan, matching, offer, booking and safety state; dedicated scan, cleaner-supply, rematching, booking/safety and profit filters expose one workstream without changing records. A scan action opens its existing review panel directly but never loads private media or submits a decision automatically. Urgent safety reports, booking finalisation and overdue follow-ups remain prioritised, while a required scan or requested revision pending for 24 hours is promoted without inferring scope or contacting the customer
- Private first-profitable-booking runway derives eight conversion stages from authoritative records—request, submitted scan, reviewed scope, sent offer, two-sided acceptance, confirmed booking, recorded outcome and current target-margin result—then names the hard bottleneck and dispatch-ready cleaner count without exposing contact data or moving money. When incomplete business readiness is the hard gate, it separately names the earliest safe operational action that can progress in parallel without bypassing that gate; later audited refunds or re-clean costs can correctly reverse the profitable milestone
- Internal quote calculator that pre-fills approved rates, includes a founder-confirmed labour on-cost percentage alongside payment fees, travel, supplies, risk contingency and additional job costs, and solves the customer total/hourly rate needed to meet the contribution-margin floor
- Evidence-based local launch-readiness form for legal identity, a valid public HTTPS website origin, pilot coverage, rates, cleaner pay, insurance, live payment handling, refunds and operating rules; the public-site gate also requires a founder-recorded ownership/deployment summary naming the exact hostname and a non-future verification date, while every incomplete area names its exact missing decisions. A guided continue action opens the long setup form at the first real missing requirement, highlights and focuses the matching field, preserves all existing entries, never fabricates a value and disappears once all seven areas are complete
- Founder-configured outward-postcode enforcement that blocks out-of-area matching, proposal use and booking readiness
- Human-reviewed matching stays closed until the room scan supplies a reviewed duration, then returns only fully screened approved cleaners whose confirmed availability can hold the job on the requested date with a start inside the customer's arrival preference
- Match results provide a feasible suggested start and finish inside the cleaner's confirmed window; proposal drafts prefill from that schedulable visit rather than the raw availability start
- UK-local scheduling: all availability, same-day cutoffs, match suggestions, quotes and confirmed assignment handoffs use the Europe/London wall clock, including British Summer Time; automated summer/winter regressions prevent a past slot being offered because of a UTC offset
- Append-only cleaner availability requests and confirmed windows with evidence notes, one-decision enforcement and auditable withdrawal; cleaner-submitted times stay pending and do not affect matching until the control desk confirms them, matching returns only approved cleaners with a future confirmed window, and proposals must fit fully inside one
- Internal draft proposals require finite, supported hours/rates/costs, freeze a complete cost breakdown and reject numeric overflow, work below minimum hours, loss-making economics or the margin floor; changed founder cost assumptions close the stale proposal across drafts, private decisions and booking
- Exact proposal start and calculated finish times, with past-date and same-day duration validation
- Sent and accepted offers temporarily reserve the cleaner's exact interval; matching moves later suggestions around those holds, and cleaner decline, withdrawal or offer expiry releases the time automatically
- Cleaner travel coverage uses explicit outward postcode districts or comma-separated postcode areas; uncovered cleaners are excluded instead of merely receiving a lower score, and direct proposals outside their stated area are rejected
- Atomic send and booking checks prevent two overlapping offers or bookings from claiming the same cleaner capacity, including concurrent send attempts and legacy booking records
- Availability is rechecked when a proposal advances, either side decides and the booking is written; withdrawal closes affected private decisions and moves the customer tracker safely back to rematching
- One-live-offer control prevents competing ready, sent or accepted proposals for the same request; a cleaner decline immediately locks the affected customer quote and allows a reviewed replacement to take priority
- Live cleaner eligibility is rechecked on the customer journey and founder dispatch queue; if approval, screening, service fit or travel coverage changes before booking, the quote action closes and the tracker returns to neutral rematching without exposing the cleaner or internal reason
- Every later proposal records the exact predecessor it replaces. A replacement cannot advance until that offer is closed or exhausted, freezes a customer-visible list of changed terms, preserves any earlier acceptance only as audit evidence and always requires a fresh private customer decision
- Audited founder proposal withdrawal requires a reason, preserves any customer acceptance record, closes both private links and returns the request to rematching. Separately, token-authorised customers can close their own unbooked enquiry from the tracker; confirmed bookings cannot be cancelled through either pre-booking control
- Proposal status gates that prevent a draft becoming ready, sent or accepted until launch checks, pilot coverage and the latest job-brief review pass
- Scan-to-quote duration protection blocks any proposal below the reviewed room-scan hours, even when its calculated contribution and margin would otherwise pass
- Review-only customer quote and cleaner opportunity drafts with explicit warnings and no send capability
- Status-gated copy-only dispatch packs pair the customer message only with the private quote link and the cleaner message only with the private opportunity link; links stay out of review-stage text, each pack names its intended recipient, completed or expired decision paths stop being sendable, and copying never contacts anyone. The proposal card exposes only explicit local audit previews, never copyable local-origin recipient URLs. Link generation fails closed until a non-local HTTPS public origin has matching deployment evidence and a non-future verification date, and that origin is frozen when the offer is sent so later settings cannot silently rewrite it
- Private customer quote-review links that keep the token out of server URLs, require name/scope/terms confirmation, lock after one decision and preserve an acceptance snapshot for the booking audit
- Customer acceptance can no longer be fabricated from the control desk; accepted and declined states come from the private quote decision flow
- Private cleaner opportunity links show only the area, reviewed scope, hazards and proposed pay; customer identity, exact address, access notes and private photos remain protected
- Customers can separately authorise the one selected cleaner to review the frozen room photos and short videos through the active private opportunity link; the exact address, identity and access details stay hidden, and preview, expired, declined, withdrawn or booked links cannot load the media
- Cleaner decisions require the application name plus scope, pay and availability confirmations, lock after one response and cannot be created from the control desk
- Founder-set customer and cleaner response windows freeze into each sent offer, are capped at the visit start, appear on private pages and drafts, and automatically close stale decisions; booking audits verify both acceptances were timely
- Latest approved landlord checklist included in the review-only cleaner opportunity draft; photos remain separately protected for deliberate review
- Structured site scope, access and hazard collection plus a read-only accepted-proposal booking audit
- Safe lead-status transitions that prevent requests or cleaners skipping required workflow stages
- Internal confirmed-booking records that require separate customer and cleaner acceptance, a profitable passed audit, one shared frozen verified public host and four remaining manual confirmations; the external payment step also requires a provider-backed reference, the exact frozen customer total and a non-future verification time after customer acceptance
- Structured confirmed-booking packs for the final address, matching postcode, access contact, arrival instructions, equipment plan and emergency instructions; duplicate bookings are rejected atomically
- Separate fragment-token customer and cleaner booking views built only from the verified public host frozen into the accepted offer and then the booking itself; a later settings or device-origin change cannot rewrite them, the customer view hides cleaner contact/pay, while the cleaner view exposes only the visit and access information needed after confirmation
- Recipient-isolated copy-only confirmed-booking handoffs pair each complete message only with its named recipient and correct frozen public booking-pack link; the control desk exposes bare links only as local audit previews, and copying never sends a message
- External payment evidence stays private to the control desk and is excluded from both booking-pack APIs; Tideway records the provider, reference, exact amount and verification time but never stores card details, credentials or performs the payment action
- Reviewed room photos, short videos and their specific notes become visible to both sides only inside the confirmed protected booking packs; media requests require the private booking token and are never cached
- Private room-media lifecycle desk classifies active, scheduled, deletion-eligible and already-deleted scans against separate founder-set inactive-enquiry and completed-booking periods; nothing is deleted automatically, and manual deletion requires a verified backup, the exact scan reference and a written append-only audit reason
- Private reschedule, cancellation, access, scope, cleaning-quality and safety request submission from either booking pack; submissions never mutate the confirmed booking, schedule or payment state automatically
- Local admin change-request queue with open, reviewing and permanently closed audit states plus a customer-visible response note
- Schedule-locked append-only job-day timeline: the private pack hides arrival until 30 minutes before the confirmed UK-local start, the server atomically rejects earlier arrival and any completion before the visit start, cleaners must confirm arrival and a safe start before recording completion, then the customer acknowledges the completed visit from their own protected booking pack
- Completed-job economics remain locked until the three job-day confirmations are present and every change or safety request is closed
- Completed-job records separate actual customer receipts, cleaner pay, labour on-costs, payment fees, travel, supplies, other costs and refunds before calculating contribution and margin; finite hours and monetary limits plus a derived-economics check prevent overflow from entering the profitable-booking runway. The initial outcome requires private, distinct customer-receipt and cleaner-payout references, a verification time after customer acknowledgement, the exact frozen customer total and at least the agreed cleaner pay, while recording never moves money
- Append-only post-completion adjustments record later refunds, re-clean work, labour on-costs, complaint resolutions and provider costs only after the external action occurred; each write atomically checks the cumulative hours, financial limits and derived economics before append. Unique evidence references prevent duplicates, optional resolved-issue links preserve context, and the effective contribution, margin and first-profitable-booking runway recalculate immediately without rewriting the original outcome

## Before public launch

1. Add the legal operator name, trading address and verified contact details.
2. Decide the cleaner engagement model with UK employment/tax advice.
3. Confirm public liability and any other required insurance.
4. Approve pricing, cleaner pay, cancellation, complaint and re-clean rules.
5. Replace local file storage and single-process rate limiting with an encrypted production database, shared abuse controls, access monitoring and unique database constraints for submission retry keys.
6. Approve the inactive-enquiry and completed-booking media periods already enforced by the local retention desk, then document the production speech-recognition provider, encrypted photo storage and the complete record-retention schedule.
7. Configure the final public HTTPS website origin, verify domain ownership and the deployed pages, then record a short evidence summary naming the exact hostname and the verification date; local, IP-address, path-bearing, insecure or unsupported origins cannot generate dispatch handoffs.
8. Add transactional email/SMS only after the sending account is approved.
9. Complete a real pilot in one small service area before making broader coverage claims.

If the server is ever bound to a public interface, set a strong `ADMIN_KEY`. Local control-desk access is automatic only when the server, request hostname and network connection are all verified as loopback and no proxy headers are present.

Set `TRUST_PROXY=true` only when Tideway is directly behind a trusted reverse proxy that removes client-supplied forwarding headers and writes its own. The local pilot ignores forwarded addresses by default, and its in-memory limits reset when the server restarts; production needs a shared persistent limiter across every app instance.
